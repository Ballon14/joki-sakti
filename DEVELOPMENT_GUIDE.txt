### Step 4.3: Auth State Management

**main.dart Implementation:**

```dart
class MyApp extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return MultiProvider(
      providers: [
        // Auth state stream
        StreamProvider<User?>(
          create: (_) => AuthService().authStateChanges,
          initialData: null,
        ),
        
        // Cart provider with user dependency
        ChangeNotifierProxyProvider<User?, CartProvider?>(
          create: (_) => null,
          update: (context, user, previousCart) {
            if (user == null) {
              previousCart?.dispose();
              return null;
            }
            if (previousCart == null || previousCart.userId != user.uid) {
              previousCart?.dispose();
              return CartProvider(userId: user.uid);
            }
            return previousCart;
          },
        ),
      ],
      child: MaterialApp(...),
    );
  }
}
```

### Step 4.4: Auth Wrapper & Role-Based Routing

**AuthWrapper Logic:**

```dart
StreamBuilder<User?>(
  stream: authService.authStateChanges,
  builder: (context, snapshot) {
    if (user == null) return LoginScreen();

      future: authService.getUserData(user.uid),
      builder: (context, userSnapshot) {
        if (userData?.isAdmin == true) {
```

## Phase 7: Order System

### Step 7.1: Order Creation

**Order Processing Logic:**

```dart
Future<String> createOrder({
  required String userId,
  required List<CartItem> items,
  required String deliveryAddress,
  required String paymentMethod,
}) async {
  // Calculate total
  double totalAmount = items.fold(0.0, (sum, item) => sum + item.totalPrice);
  
  // Create order document
  final order = OrderModel(...);
  DocumentReference docRef = await _firestore.collection('orders').add(order.toMap());
  
  // Update product stocks
  for (var item in items) {
    await _firestore.collection('products').doc(item.product.id).update({
      'stock': FieldValue.increment(-item.quantity),
    });
  }
  
  return docRef.id;
}
```

## Key Technical Decisions

### 1. **Why Provider over Bloc/Riverpod?**
- Simpler learning curve
- Sufficient for app complexity
- Good integration with Firebase streams

### 2. **Why Firebase?**
- No backend server needed
- Real-time data sync
- Built-in authentication
- Scalable infrastructure

### 3. **Why Firestore over Realtime Database?**
- Better query capabilities
- Offline support
- Document-based structure fits e-commerce

### 4. **Architecture Pattern**
- Service layer separates business logic
- Providers handle state
- Clear separation of concerns

### 5. **Web-First Responsive Design**
- Mobile-first approach
- Progressive enhancement for web
- Single codebase for all platforms
